# デジタル庁 ダッシュボードデザインガイドブック スキル

出典：デジタル庁「ダッシュボードデザインの実践ガイドブック」2026年3月31日更新
GitHub: https://github.com/digital-go-jp/policy-dashboard-assets

## 0. ダッシュボードの2類型

| 類型 | 目的 | 前提知識 | 操作 |
|------|------|---------|------|
| 提示型 | 現状把握・異常検知・行動判断 | 一般的な知識 | 単純 |
| 探索型 | 詳細分析・差分発見・源流特定 | 特定ドメイン知識 | 複雑 |

**本スキルは提示型を中心に扱う。**

---

## 1. 作成の3ステップ

```
要件の整理 → プロトタイピング → 実装
```

### Step 1：要件の整理（5W1H）
- **Why**：ダッシュボードの目的は何か？
- **What, so What**：どんな行動のために、どの情報を知るべきか？
- **Who, When, Where**：誰が、いつ、どこで見るのか？
- **How**：求められる機能・データ項目・更新頻度は？

**制約の確認チェックリスト**
- 情報の種類が多すぎないか
- 伝える内容が複雑すぎないか
- データに欠損・更新頻度不足はないか
- 指標同士を比較できるか

### Step 2：プロトタイピング
1. 必要な情報を一覧化し、グラフ候補を選定
2. 骨格となるレイアウトを先に固める（表層の詰めはその後）
3. 関係者にフィードバックをもらい改善

**プロトタイプ設計の4原則**
- 構造を伝える（全体→部分の階層）
- 複雑な操作を要求しない
- 適切な情報量にする
- 比較対象を提供する

### Step 3：実装
- Power BIのデザインテンプレートを活用
- チェックリストで最終確認

## 2. レイアウト設計

**基本原則：左上から右下へ、全体→詳細の流れ**

```
┌─────────────────────────────────────┐
│ 全体指標（大きく・目立つ）           │
│ 数値テキスト    構成比    フィルター  │
├─────────────────────────────────────┤
│ 中程度の詳細                         │
│ 時系列グラフ    分類別グラフ          │
├─────────────────────────────────────┤
│ 最も詳細な情報（表・細かい分解）      │
└─────────────────────────────────────┘
```

**レイアウトグリッド：16:9画面を縦横2〜6分割**

**フィルターの配置**
- フィルターは画面の上部か左部
- フィルターの影響を受ける情報はその下か右

---

## 3. グラフの種類と選び方

| グラフ | 値の表現 | 伝える情報 | 使う場面 |
|--------|---------|-----------|---------|
| 折れ線グラフ | 位置 | 時間変化・傾向 | 横軸が時間の場合 |
| 棒グラフ | 長さ | 数量比較 | 分類ごとの数量比較 |
| 面グラフ | 位置・面積 | 時間変化・傾向・構成比 | 複数系列の積み上げ |
| 円グラフ | 角度・面積 | 構成比 | 全体総量が明確な場合のみ |

**選択の判断基準**
- 時間変化→折れ線グラフ
- 数量比較→棒グラフ（まず検討）
- 構成比→円グラフ（総量が明確な場合）、棒グラフの方が正確なことが多い
- 時間変化＋構成比→面グラフ

## 4. カラーパレット（デジタル庁デザインシステム準拠）

7系統のカラーパレット（Power BIテンプレートに組み込み済み）

| パレット | 主な用途 |
|---------|---------|
| Blue（#3460FB系） | 標準・メイン |
| Light Blue（#008BF2系） | サブ |
| Cyan（#00A3BF系） | 補助 |
| Green（#259D63系） | 肯定・成功 |
| Orange（#FB5B01系） | 強調・警告 |
| Red（#FE3939系） | 否定・エラー |
| Solid Gray（#4D4D4D系） | ニュートラル |

**コントラスト比の基準**
- グラフ色面と背景：3:1以上
- テキスト色と背景：4.5:1以上
- 満たせない場合は数値を色面近くに記載

---

## 5. グラフ設計の原則

### 知りたいことを知れる
- シンプルにする（不要な情報を取り除く）
- 意味のある順列にする（数量の大小順など）
- 強弱をつける（重要箇所を色・サイズで強調）
- 待たせすぎない（読み込み・操作レスポンス）

### 誤解を生まない
- わかりやすく表記する（前提知識不要の表現）
- データを定義する（対象・意味・更新日を明示）
- 表現を歪曲しない（軸の範囲を恣意的に設定しない）
- メタ情報を記載する（データソース・更新日・注釈）

---

## 6. Do's / Don'ts（必ず確認）

### Do ✅
- 全体指標と詳細グラフを両方表示する
- グラフ項目の並び順に意味を持たせる（数量大小順など）
- 不要な要素は削除する（グリッド・枠線は最小限）
- グラフに使う色数を1〜5色に絞る
- タイトルにグラフの内容とデータ種別を表記する（例：「国産自動車の出荷台数（月次推移）」）
- グラフと凡例を隣接させ、順序を対応づける
- **棒グラフの原点は必ず0にする**
- データ更新日を記載する
- 色以外でも系列を識別できるようにする（形・模様）

### Don't ❌
- 詳細の要素だけで全体指標がない
- 意味もなくあいうえお順で並べる
- 3D表現・ドロップシャドウなど不要な装飾
- 多色使い（注目点がわからなくなる）
- タイトルが冗長・何を表すか不明
- 凡例がグラフから離れている・順番が逆
- 棒グラフの原点が0以外（数量が誇張・矮小化される）
- 色だけで分類を識別している

## 7. チェックリスト（実装後の確認）

### 利用者の体験
- [ ] 判断・行動に必要な情報が提供できているか
- [ ] 全体像を捉えるための基準指標が掲載されているか
- [ ] 本当に必要な情報だけに絞れているか
- [ ] 少ない操作で目的が達成できるか

### グラフ設計
- [ ] 重要な数値に優先的に目が向くか
- [ ] グラフと凡例が隣接・対応づけされているか
- [ ] グラフの色数が1〜5色か
- [ ] 棒グラフの原点は0か
- [ ] タイトルにデータ種別（月次推移・累計など）が入っているか
- [ ] 不要な装飾（3D・シャドウ）を使っていないか
- [ ] データ更新日が記載されているか

### アクセシビリティ
- [ ] 色だけでグラフを識別していないか
- [ ] グラフに代替テキスト（alt）を付与しているか
- [ ] 一般公開の場合、データファイル（Excel/CSV）を公開しているか

## 8. 実装ツール（Power BI用）

- **デザインテンプレート**：7色カラーパレット適用済み。折れ線・棒・面・円・積み上げ・複合グラフ収録
- **レイアウトグリッド**：16:9画面を縦横2〜6分割
- **カラーパレット**：Blue/LightBlue/Cyan/Green/Orange/Red/SolidGray の7種（.pbit形式）

**使い方**
1. カラーパレット（.pbit）を選んでPower BIで開く
2. デザインテンプレートからグラフをコピー＆ペースト
3. レイアウトグリッドに沿ってサイズ・位置を調整

## 9. EmTripでの活用シーン

- **OPePeダッシュボード**：セッション・参加者・チェックイン数の可視化
- **ココロクロス**：登録者数・継続率・会話数のKPIダッシュボード
- **マツリクロス**：イベント参加者・満足度・地域別分布
- **投資家向け資料**：売上推移・ユーザー成長・KPI達成率

**注意**：EmTripのブランドカラー（#4653A2・#ED6D46）とデジタル庁パレットを混在させる場合は、EmTripカラーを優先しつつ、グラフの可読性（コントラスト比3:1以上）を維持すること。
