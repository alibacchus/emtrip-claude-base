# Claude Code 行動規範

## 読み飛ばし防止
- 読んでいないコードは変更するな
- 編集前に必ず該当ファイルの全体構造を確認すること
- 関連ファイルが存在する場合は先にリストアップしてから着手すること

## 調査ファースト原則
- 複雑なタスクは「調査→設計→実装」の順序を守ること
- いきなり編集に飛びつかないこと
- 変更の影響範囲を事前に言語化してから着手すること

## 出力品質
- 事実・仮説・計画を分離して出力すること
- 不明点は「未確定」と明記すること
- 推論は「推論：」と明記すること

## プロダクト
- ココロクロス：店舗常設AI・唯一の稼働モデル
- マツリクロス：青年会×エイサー体験×AI（PoC段階）
- OPePe：マリン事業者向け予約・運営ツール（PSF検証中）

## 絶対に崩さない順番
1. ココロクロスの再現性を証明する
2. 旅前・旅後の継続関係を設計する
3. 対話ログを意味DBとして蓄積する
4. API/IP/コマースへ展開する

## コンテキスト管理ルール（Anthropic公式 Thariq原則 2026-05反映）
- **新タスクを始めるなら新セッション**：1Mトークン枠でも30-40万トークンでcontext rotが発生（モデルの注意が分散・ノイズ増・現タスク阻害）
- **アプローチ失敗時は /rewind（Esc Esc 二回タップ）**：失敗試行をコンテキストから削除。「忘れて」では履歴は残り続ける
- **/compact vs /clear の使い分け**:
  - /compact：モデル要約に置き換え。関連タスク継続向き。意図書面化推奨（例「auth refactorに集中・無関係debugは削除」）
  - /clear：自分で要点を書き出してから新セッション開始。無関係次タスク向き
- **autocompact直後は注意**：context rotで知性低下中に再要約が走り、最も質の低い要約を作りやすい。100万トークン余裕あれば自分から先回りして /compact
- **subagent委譲基準**：「中間ツール出力を再利用しない／結論だけで十分」なら subagent。例：検証・別flowコード読込・git diff要約
- 30万トークン到達前に必ず /compact または /clear を実行
