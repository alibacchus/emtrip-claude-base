## Claude自己分析の信頼性ルール（2026-05-12 追加）

**Claude (このAI自身) に「なぜハルシネーションした？」「なぜ間違えた？」と聞くと、その答え自体が事後合理化的ハルシネーションを含むことがある**（2026-05-12 石垣Mac 事例で観測：「CLAUDE.mdに『ここいく』と書かれていた」と虚偽報告した）。

### Claude自身が守るべきメタルール
1. 「なぜそう書いたか」を問われたとき、**メモリに無い場合は「実ファイルを Read で確認してから答える」** が正解
2. 「資料の◯◯にこう書いてあった」と主張する前に、**該当ファイルを実際に Read** する
3. 自分の出力ミスを正当化するために「ソースにこう書いてあった」と創作するのは**重大違反**
4. 自己批判を求められても、**事実確認できない部分は「推測：」「確認不能：」と明記**

### ユーザーが守るべき運用ルール（与那国・石垣・将来のチームメンバー）

| 場面 | やってはいけない | 正しい対処 |
|---|---|---|
| Claude が固有名詞を間違えた | Claude に「なぜ間違えた?」と聞いて答えを信じる | `grep` で実ファイル確認 |
| Claude が「資料にこう書いてあった」と主張 | そのまま信じる | 該当ファイルを自分で開いて確認 |
| Claude が「私のミスは◯◯が原因」と分析 | 採用してそのまま対策を作る | 事実ベースで再評価（grep / 実ファイル Read） |
| Claude が「対策はこうすべき」と提案 | そのまま実装する | 自分のロジックで検証してから |

**結論**: Claude は **「事後合理化が得意な確率モデル」**。デバッグ・原因分析を Claude 単体に任せると、二重のハルシネーションが起きる。**重要な判断は「Claude が言った」より「実ファイルが何を言っているか」で決める**。

## コンテキスト管理ルール（Anthropic公式 Thariq原則 2026-05反映）
- **新タスクを始めるなら新セッション**：1Mトークン枠でも30-40万トークンでcontext rotが発生（モデルの注意が分散・ノイズ増・現タスク阻害）
- **アプローチ失敗時は /rewind（Esc Esc 二回タップ）**：失敗試行をコンテキストから削除。「忘れて」では履歴は残り続ける
- **/compact vs /clear の使い分け**:
  - /compact：モデル要約に置き換え。関連タスク継続向き。意図書面化推奨（例「auth refactorに集中・無関係debugは削除」）
  - /clear：自分で要点を書き出してから新セッション開始。無関係次タスク向き
- **autocompact直後は注意**：context rotで知性低下中に再要約が走り、最も質の低い要約を作りやすい。100万トークン余裕あれば自分から先回りして /compact
- **subagent委譲基準**：「中間ツール出力を再利用しない／結論だけで十分」なら subagent。例：検証・別flowコード読込・git diff要約
- 30万トークン到達前に必ず /compact または /clear を実行
